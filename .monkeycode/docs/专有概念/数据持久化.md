# 数据持久化

所有用户数据存储在浏览器 IndexedDB 中，采用用户隔离 + 词库隔离的双层 key 设计，支持跨版本自动迁移。

## 什么是数据持久化？

应用使用 IndexedDB 替代 localStorage，提供更大的存储容量、结构化的 Object Store 组织，以及 `onupgradeneeded` 事件驱动的版本迁移能力。

## 数据库 Schema

**数据库**: `cet4_study` v3

### Store: `profile`

| Key | Value 类型 | 说明 |
|-----|-----------|------|
| `active` | string | 当前活跃用户的 `uid` |
| `user:{uid}` | `UserProfile` | 用户档案 `{username, uid, activeBank, createdAt}` |

### Store: `data`

| Key | Value 类型 | 说明 |
|-----|-----------|------|
| `data:{uid}:{bank}` | `StudyData` | 某用户某词库的学习数据 |
| `data:{uid}` | `StudyData` | v2 遗留格式（迁移后保留） |
| `study_data` | StudyData | v1 遗留格式 |

### Store: `settings`

| Key | Value 类型 | 说明 |
|-----|-----------|------|
| `settings:{uid}:{bank}` | `UserSettings` | 某用户某词库的设置 |
| `settings:{uid}` | UserSettings | v2 遗留格式 |
| `theme:{uid}` | string | 主题偏好 `dark|light`（全局） |

## 迁移历史

### v1 → v2

将无用户隔离的全局 key 迁移为带 `uid` 的 key：

```
v1: data/study_data  →  v2: data/data:{uid}
v1: settings/app_settings  →  v2: settings/settings:{uid}
v1: profile/username  →  v2: profile/user:{uid}.username
```

### v2 → v3

将无词库隔离的 key 拆分为按词库独立的 key，并将主题字段从 settings 中独立出来：

```
v2: data/data:{uid}      →  v3: data/data:{uid}:cet4
v2: settings/settings:{uid}  →  v3: settings/settings:{uid}:cet4
                              +  settings/theme:{uid}  (主题独立存储)
```

迁移后 `.theme` 字段从 `settings` 对象中删除，避免保存到词库相关 key 中。

## 不变量

1. **uid 不可变**: 生成后永久复用，通过 `profile/active` 索引
2. **迁移只追加不修改**: 旧版本迁移代码保留不动，新版本只追加
3. **迁移先于加载**: `openDB()` → `migrateV1toV2()` → `loadAll()` → `migrateKeys()` → `loadWords()`
4. **内存同步**: 迁移后立即更新内存中的 `data` 和 `settings`，避免后续操作丢失迁移数据
5. **静默失败**: 写入失败只 `console.warn`，不抛出异常阻断 UI

## 存储容量

IndexedDB 没有像 localStorage 那样的 5MB 硬限制。浏览器通常在总磁盘空间的一定比例内允许存储（通常为数 GB）。对于本应用的 JSON 词库和文本数据，完全足够。
